# ATLAS — New IP Recipe

How to add a new IP to ATLAS with cycle-accurate FL/RTL co-simulation.
Validated with four demo IPs in 2026-05-21 that each reached 100% honest
PASS on the first sim after ~1.25 SSOT iterations.

## TL;DR — 5 files, ~300 lines of hand-work

```
<ip>/
  yaml/<ip>.ssot.yaml          # 200-330 lines — function_model + cycle_model + scenarios
  rtl/<ip>.sv                  # 50-55 lines  — hand-written RTL
  rtl/rtl_contract.json        # 25-40 lines  — bridges SSOT outputs to RTL ports
  list/<ip>.f                  # 2 lines      — filelist
```

Everything else (FunctionalModel, cocotb test, scoreboard, manifest, goals,
~6000 lines) is auto-generated by ATLAS emit scripts.

## The three commands

```bash
python3 workflow/fl-model-gen/scripts/emit_fl_model.py            <ip> --root .
python3 workflow/fl-model-gen/scripts/emit_equivalence_goals.py   <ip> --root .
python3 workflow/tb-gen/scripts/emit_goal_scoreboard_cocotb.py    <ip> --root .
cd <ip>/tb/cocotb && python3 test_runner.py
```

A successful run prints:

```
[CL_COSIM] <ip> cycle-accurate FL/RTL co-sim: N/N match (100.0%)
TESTS=1 PASS=1 FAIL=0
scoreboard fail: 0 / N
```

## Pick the recipe that matches your IP

| Pattern | When to use | Example | Demo file |
|---|---|---|---|
| **Output mirror** | Single register drives an output bus | GPIO output, status mirror | `apb_gpio_demo/` |
| **CSR + pulse output** | Register write generates a 1-cycle output signal | UART TX strobe, IRQ fire | `atcuart_mini/` |
| **Multi-register combinational** | Output depends on two or more registered values | Equality compare, threshold | `apb_compare/` |
| **Input-driven counter / FSM** | External input drives state evolution | Pulse counter, debouncer | `apb_pulse_counter/` |

## Recipe 1: Output mirror — `apb_gpio_demo`

A single register's value mirrors to an output port.

### Minimal SSOT structure

```yaml
function_model:
  state_variables:
    # IMPORTANT: pick a name DIFFERENT from any register name to avoid
    # env collision (state.NAME would be overwritten by registers.NAME).
    - { name: gpio_data, source: registers.DATA.data, reset: 0,
        description: "Backing storage; drives output" }
  transactions:
    - id: FM_WRITE
      preconditions: ["psel == 1", "penable == 1", "pwrite == 1", "paddr == 0"]
      output_rules:
        - { name: pready_rule,    port: PREADY,  expr: 1, width: 1 }
        - { name: pslverr_rule,   port: PSLVERR, expr: 0, width: 1 }
        - { name: gpio_out_rule,  port: gpio_out, expr: "pwdata & 0xff", width: 8 }
      state_updates:
        - { name: gpio_data, expr: "pwdata & 0xff", width: 8 }
    - id: FM_IDLE
      preconditions: ["pwrite == 0 or psel == 0 or penable == 0"]
      output_rules:
        - { name: gpio_out_rule, port: gpio_out, expr: "gpio_data & 0xff", width: 8 }

cycle_model:
  cosim: true                # ← opt-in to cycle-accurate CL co-sim
```

### Common gotcha — name collision

If you name a `state_variable` the same as a register (e.g. both `DATA`),
the FL env builder overwrites the state with the register's current value
(which hasn't been updated). Always name state vars distinctly:

```yaml
# ❌ WRONG — DATA shadows itself
state_variables:
  - { name: DATA, source: registers.DATA }
registers:
  - name: DATA

# ✅ RIGHT — distinct names
state_variables:
  - { name: gpio_data, source: registers.DATA.data }
registers:
  - name: DATA
```

## Recipe 2: CSR + pulse output — `atcuart_mini`

Register write generates a transient output (e.g. UART TX serial pulse).

### Key SSOT pattern

```yaml
function_model:
  state_variables:
    - { name: thr_data,   source: registers.THR.data, reset: 0 }
    - { name: tx_busy_q,  reset: 0,
        description: "Pulses high for 1 cycle after THR write" }
  transactions:
    - id: FM_THR_WRITE
      preconditions: ["psel == 1", "penable == 1", "pwrite == 1", "paddr == 0"]
      output_rules:
        - { name: tx_serial_rule, port: tx_serial, expr: "pwdata & 1", width: 1,
            description: "Mirrors PWDATA bit 0 during write cycle" }
      state_updates:
        - { name: thr_data,  expr: "pwdata & 0xff", width: 8 }
        - { name: tx_busy_q, expr: 1, width: 1 }
    - id: FM_IDLE
      preconditions: ["pwrite == 0 or psel == 0 or penable == 0"]
      output_rules:
        - { name: tx_serial_rule, port: tx_serial, expr: 1, width: 1,
            description: "Idle high" }
      state_updates:
        - { name: tx_busy_q, expr: 0, width: 1 }
```

The key insight: the pulse output is a `tx_serial` rule whose expression
is *different* in the active transaction vs the idle transaction. ATLAS's
SSOT-driven `step()` picks the right transaction per cycle from the
preconditions.

## Recipe 3: Multi-register combinational — `apb_compare`

Output is a combinational function of two or more registered values.

### Key SSOT pattern

```yaml
function_model:
  state_variables:
    - { name: value_q,     source: registers.VALUE.data,     reset: 0 }
    - { name: reference_q, source: registers.REFERENCE.data, reset: 0 }
  transactions:
    - id: FM_WRITE
      preconditions: ["psel == 1", "penable == 1", "pwrite == 1"]
      output_rules:
        - { name: match_rule, port: match_o, width: 1,
            # IMPORTANT: use post-write values for combinational compare
            expr: "1 if ((pwdata if paddr == 0 else value_q) ==
                           (pwdata if paddr == 4 else reference_q)) else 0" }
      state_updates:
        - { name: value_q,     expr: "pwdata if paddr == 0 else value_q",     width: 32 }
        - { name: reference_q, expr: "pwdata if paddr == 4 else reference_q", width: 32 }
    - id: FM_IDLE
      preconditions: ["pwrite == 0 or psel == 0 or penable == 0"]
      output_rules:
        - { name: match_rule, port: match_o, expr: "1 if value_q == reference_q else 0", width: 1 }
```

### Why post-write values matter

RTL drives `match_o` *combinationally* from the registered values after
the rising edge. The FL must mirror this — during a write, the output
sees the *new* value, not the stale one. The conditional in
`(pwdata if paddr == 0 else value_q)` captures this.

## Recipe 4: Input-driven counter — `apb_pulse_counter`

External input signal drives state evolution.

### Key SSOT pattern

```yaml
io_list:
  interfaces:
    - name: pulse_io
      ports:
        - { name: pulse_in,  width: 1, direction: input,
            description: "Increments counter when high" }

function_model:
  transactions:
    - id: FM_CLEAR
      # CLEAR has priority over PULSE — list it first.
      preconditions: ["psel == 1", "penable == 1", "pwrite == 1", "paddr == 4"]
      state_updates:
        - { name: count_q, expr: 0, width: 8 }
    - id: FM_PULSE
      preconditions: ["pulse_in == 1",
                      "(psel == 0 or penable == 0 or pwrite == 0 or paddr != 4)"]
      state_updates:
        - { name: count_q, expr: "(count_q + 1) & 0xff", width: 8 }
    - id: FM_IDLE
      preconditions: ["pulse_in == 0",
                      "(pwrite == 0 or psel == 0 or penable == 0 or paddr != 4)"]
      state_updates:
        - { name: count_q, expr: count_q, width: 8 }
```

### Gotcha — heuristic stimulus driving input pins

ATLAS's heuristic stimulus generator may drive boolean-looking input
pins to 1 by default (e.g. `pulse_in`). For scenarios where the input
should be low, explicitly pin it in `stimulus_machine_spec.assign`:

```yaml
- id: SC3
  name: "Clear via APB write"
  stimulus_machine_spec:
    timeline:
      - { assign: { pulse_in: 0 } }   # ← explicitly hold low so CSR clear is observable
      - { csr_write: { offset: 0x4, data: 0x1 } }
      - { assign: { pulse_in: 0 } }
```

## Common SSOT sections (used by every IP)

```yaml
top_module:
  name: <ip>
  file: rtl/<ip>.sv
  version: "1.0"
  type: peripheral
  description: "..."
  target: { technology: generic, clock_freq_mhz: 50 }

io_list:
  clock_domains:
    - { name: pclk, ports: [{ name: PCLK, width: 1, direction: input }] }
  resets:
    - { name: presetn, polarity: active_low,
        ports: [{ name: PRESETn, width: 1, direction: input }] }
  interfaces:
    - { name: apb_slave, type: APB-Lite, role: slave, ports: [...] }
    - { name: <ip>_output, type: custom, ports: [...] }

registers:
  config: { register_width: 32, addr_width: 4 }
  register_list:
    - { name: <REG>, offset: 0x0, access: rw, reset: 0, fields: [...] }

rtl_contract:
  transaction: <primary FM>
  clock: PCLK
  reset: PRESETn
  reset_active: low
  input_map: { psel: PSEL, penable: PENABLE, pwrite: PWRITE,
               paddr: PADDR, pwdata: PWDATA }

test_requirements:
  scenarios:
    - id: SC1
      stimulus_machine_spec:
        timeline:
          - { csr_write: { offset: 0x0, data: 0x55 } }
```

## Hand-written RTL skeleton

```systemverilog
`timescale 1ns/1ps

module <ip> #(parameter int APB_DATA_WIDTH = 32) (
    input  logic                       PCLK, PRESETn,
    input  logic [3:0]                 PADDR,
    input  logic                       PSEL, PENABLE, PWRITE,
    input  logic [APB_DATA_WIDTH-1:0]  PWDATA,
    output logic [APB_DATA_WIDTH-1:0]  PRDATA,
    output logic                       PREADY, PSLVERR,
    // IP-specific I/O...
);
    logic [...] <state>_q;

    wire access_phase = PSEL & PENABLE;
    wire write_<reg>  = access_phase & PWRITE & (PADDR == 'h0);

    always_ff @(posedge PCLK or negedge PRESETn) begin
        if (!PRESETn) <state>_q <= '0;
        else if (write_<reg>) <state>_q <= PWDATA[...];
    end

    assign PREADY  = 1'b1;          // zero-wait-state slave
    assign PSLVERR = 1'b0;
    assign PRDATA  = ...;           // mux per offset
    // IP-specific output assignments...
endmodule
```

## SSOT YAML gotchas (from real experience)

| Gotcha | Symptom | Fix |
|---|---|---|
| `[0]` in unquoted description | `yaml.safe_load` parse error | Quote the string: `description: "..."` |
| Em-dash `—` in unquoted text | Parse error on some setups | Use ASCII `-` or quote string |
| State var name == register name | output_rule evaluates to 0 | Rename state var (e.g. `gpio_data` not `DATA`) |
| `reset: "all-ones"` | Validator blocker | Use numeric: `reset: 15` or `reset: (1 << WIDTH) - 1` |
| Missing `output_rules` on a transaction | Validator blocker | Add at least `pready` / `pslverr` rules |

## Cycle-accurate co-sim opt-in

For state-rich IPs (multi-cycle FSMs, counters, accumulators):

```yaml
cycle_model:
  cosim: true                    # ← enables FL.step() per-cycle co-sim
```

For state-accumulating IPs (round-robin arbiter, etc.) that must
preserve state across consecutive scenarios:

```yaml
cycle_model:
  cosim: true
  state_accumulating: true       # ← disables per-goal reset
```

Without `cosim: true`, ATLAS still verifies via single-shot `FL.apply()`
but cannot catch state-evolution bugs cycle-by-cycle.

## When `step_unresolved` happens

If `FL.step()` references a name not in env, the FL returns a graceful
`step_unresolved` result and CL co-sim skips that cycle. Common causes:

- SSOT output_rule uses a decoded combinational signal not in
  state_variables (e.g. `branch_taken`, `is_store` in a CPU). Solution:
  add it as a state var or input.
- Typo in an expression. Solution: re-emit and check FL self-check.

A `step_unresolved` ratio above ~10% means the SSOT is incomplete; check
the `[CL_COSIM] match` line — if it's below 80%, investigate.
